# 공수 계산 — 기능 기획서 (초안)

> Teamlog 업무 기록(`WorkLog`)·`taskCode` 설계를 전제로 한 **관리자용「공수 계산」메뉴** 도입 계획입니다.  
> 코드·빌드 작업 전, 요구사항·집계 규칙·확장 아이디어를 문서로만 고정합니다.

---

## 1. 배경과 목적

### 1.1 업무 기록이 의미하는 것

- **기록 단위**: 팀원이 일별로 남기는 **업무 한 줄**이 `WorkLog`이며, `duration`(시간)·`count`(건수)·`category`(대분류·소분류)·`status`·`workIndicator` 등을 포함한다.
- **동일 업무 연속성**: **동일 업무**를 여러 날에 나누어 적는 경우, 시스템은 **`taskCode`(업무 단위 ID)** 로 같은 업무임을 식별한다. 이때 **소요시간은 여러 줄의 `duration`을 합산**하여 “한 업무의 총 투입 시간”으로 본다. (진행중/완료가 섞여 있어도 같은 `taskCode`면 합산 대상에 포함하는 것이 공수 관점에서 일반적이다.)
- **분류**: `category`는 `"대분류"` 또는 `"대분류 > 소분류"` 문자열로, 대·소분류 통계의 축이 된다.

### 1.2 이 메뉴가 답해야 할 질문

- “이번 달(또는 분기) **품질검증** 소분류에 **얼마나 시간이 들어갔고**, **건수는 몇 건**이며, **건당 평균적으로 얼마나 걸렸는가**?”
- “**표준 공수**(또는 목표 공수) 대비 **실제**는 어떤가?” (표준값은 별도 정의·입력이 필요)
- “그 업무를 **누가 얼마나** 맡았는가?” (인원별 실 투입 시간)

---

## 2. 집계 규칙 (반드시 합의할 것)

### 2.1 `taskCode`가 있는 행

- **공수(시간) 합산**: `memberId` + `taskCode`가 같으면 **물리적으로 여러 줄이어도 “하나의 업무”**로 보고, 해당 그룹의 **`duration` 합**을 그 업무의 총 소요시간으로 사용한다.
- **기간**: 화면에서 “시작일~종료일” 또는 “기록이 있는 날 수”를 보여줄 수 있도록, 그룹 내 `date`의 최소·최대 또는 고유 일자 개수를 함께 계산한다. (상세는 UI 섹션 참고)
- **건수 `count`**: 현재 로그 설계상 일자별 줄마다 건수가 있다. 공수 메뉴에서 “**업무 건수**”를 어떻게 정의할지 선택이 필요하다.  
  - **안 A**: `taskCode` 그룹당 **대표 1건**만 세지 않고, **각 일자 줄의 `count` 합**을 “누적 건수”로 본다. (작업 물량이 날마다 쌓이는 모델에 맞음)  
  - **안 B**: `taskCode`당 **1건의 업무**로 보고 `count`는 **최종 줄만** 또는 **max** 등 규칙을 둔다.  
  → **권장**: 제품 기본은 **안 A(합산)** 을 노출하고, 필요 시 “줄 단위 건수”와 “업무 단위(taskCode) 건수”를 구분해 표시할 수 있게 한다.

### 2.2 `taskCode`가 없는 행 (구 데이터 등)

- **줄 단위**로만 집계한다. (합산 그룹 없음)
- 대·소분류 통계에는 **그대로 `duration`·`count`를 누적**한다.

### 2.3 상태 `status` (완료 / 진행중 / 취소)

- **공수 합산에서 제외할지 여부**:  
  - **권장 기본**: `취소`는 합산에서 **제외**. `진행중`은 **포함**(이미 투입된 시간으로 간주).  
- 화면에서 **필터**로 “진행중 포함/제외”, “취소 제외(고정)” 등을 제공하면 현장 혼선을 줄일 수 있다.

### 2.4 팀·기간 범위

- **팀**: 현재 선택된 팀(관리자 컨텍스트) 소속 팀원의 로그만 사용한다.
- **기간**: 월·분기·연·사용자 지정 기간 등을 선택한다. (최소는 **월** 단위부터)

## 4. 화면에 보여줄 지표 (1차 범위)

### 4.1 대분류별 요약

| 지표 | 정의(초안) |
|------|------------|
| 전체 소요시간 | 해당 대분류에 속하는 모든 로그(및 taskCode 합산 반영 로직 적용 후)의 `duration` 합 |
| 전체 진행 건수 | 위 2.1에 따른 **건수 정의**에 따른 건수 합(또는 업무 단위 건수) |
| 건당 소요시간 | `전체 소요시간 ÷ 전체 진행 건수` (건수 0이면 표시 제외 또는 “-”) |
| 표준 공수 | **별도 마스터 데이터** (아래 5장). 대분류 또는 대분류×소분류 단위로 목표 시간/건당 시간 등을 저장해 두고 대비 |
| 참여 인원 수 | 해당 대분류에 실제로 `duration > 0` 등으로 기여한 **고유 memberId 수** |
| 인원별 실 업무 시간 | 팀원별로 해당 대분류에 배분된 시간 합 (배분 규칙: 아래 4.3) |

### 4.2 소분류별 요약

- 대분류와 **동일한 지표 세트**를 **소분류(또는 `대분류 > 소분류`) 단위**로 표시한다.
- UI: 대분류를 펼치면 소분류 테이블이 나오는 **트리·드릴다운**이 읽기 좋다.

### 4.3 인원별 실 업무 시간 (배분 규칙)

- **taskCode 그룹**에 여러 명이 관여한 경우는 **현재 스키마만으로는 한 업무에 여러 사람이 찍혀 있지 않다**(한 줄에 `memberId` 하나). 따라서 **1차**는 “**줄에 적힌 사람에게 그 줄의 시간을 귀속**”한다.
- **동일 taskCode가 여러 명에게 나뉘어 적힌 경우**(이어하기·복제로 각자 날짜별 기록)에는, 그룹 합산 시간을 **어떻게 인원에게 나눌지** 정책이 필요하다.  
  - **1차 권장**: “**줄 단위 실제 기록**”만 인원 합산에 사용하고, **taskCode 합산 총시간**은 **분류 합계**에는 쓰되 **인원별에는 줄 단위 배분**만 쓴다(중복 없음).  
  - **2차(옵션)**: “한 업무의 총시간을 참여자 수로 균등 분배” 같은 가상 배분 — 현업 합의 후 옵션으로.

### 4.4 “표준 공수”와의 비교

- 표준값이 있으면 **편차(±시간, ±%)** 를 같은 표에 컬럼으로 둔다.
- 표준이 없는 분류는 공란 또는 “미설정”.

## 11. 구현 반영 (앱, 2026)

- **메뉴**: 관리자 사이드바·관리 홈 카드에서 **「공수 계산」** (`effort_calculation` 탭).
- **집계**: `@/utils/effortCalculation.ts` — taskCode 그룹 **건수는 안 A(일자별 count 합산)**; **취소 제외**, **진행중**은 옵션(체크 시 제외).
- **소분류 기준건당·상태(§12)**: 유닛 건당(h) **중앙값**을 기준건당으로 사용; **IQR 상단** 기준으로 집계 건당을 **정상 / 주의 / 이상**으로 분류(유닛 &lt; 5건은 `집계중`). 구현: `classifyPerCaseUpperIqrTiers`, `collectPerCasesBySub`, `buildBaselineStatsMaps`(median).
- **UI**: **대분류 → 소분류 드릴다운** 테이블, **화면 안내** 다이얼로그(초보자용 설명), 소분류 행 **로그** → Sheet에서 유닛 목록·**상세** 시 `logsForMergedUnit`으로 일자별 `WorkLog` 표시. (수동 목표 입력 패널은 탭에서 제거; 유틸 `loadEffortStandards` 등은 향후 다른 화면용으로 유지 가능.)
- **인원별**: 대분류 펼침 하단에 **줄 단위** 요약(기존과 동일).

## 12. 구현 가이드라인 (통계 원칙 · `WorkLog` 정합)

> 아래는 `src/types/workLog.ts`의 `WorkLog`·`WorkStatus`·`taskCode` 정의와 맞춘 **관리자 [공수 계산]** 핵심 집계·통계 규칙이다. 구현은 `@/utils/effortCalculation.ts`와 공수 탭 UI에 반영한다.

### 12.1 업무 유닛(Aggregation) — `taskCode` 기반

| 항목 | 규칙 |
|------|------|
| 유닛 키 | 동일 `memberId` + 동일 `taskCode`를 가진 행 → **하나의 업무 유닛** (`MergedEffortUnit`). |
| `taskCode` 없음 | 행 1줄 = 유닛 1개(합산 그룹 없음). |
| `duration` | 유닛 내 **합(Sum)**. |
| `count` | **안 A(기본)**: 유닛 내 일자별 `count`의 **합**. |
| 분류 | 유닛의 대·소분류는 그룹 내 **최신 `date` 로그**의 `category` 문자열에서 파싱(`대분류` / `대분류 > 소분류`). |
| `status` | **`취소`는 집계 제외**. **`진행중`은 기본 포함**(화면 옵션으로 제외 가능). **`완료` 포함**. |

### 12.2 소분류별 기준건당 공수 — Robust (중앙값)

- 각 **소분류 키**(`major` + `sub`)에 대해, 해당 소분류에 속한 **각 유닛**의 **건당(h)** = `유닛 총 소요시간 ÷ 유닛 총 건수` (건수 0 유닛은 표본에서 제외) 리스트를 만든다.
- **기준건당 공수** = 그 리스트의 **중앙값(Median)**. (단순 평균은 이상치에 민감하므로 기본 기준으로 쓰지 않는다.)

### 12.3 IQR 기반 상태(상·중·하한) — 건당(h) 상단 꼬리

- **동일 소분류** 내, 위 12.2의 건당(h) 표본이 **5건 미만**이면 **판정 불가(데이터 수집 중)**.
- **5건 이상**일 때만 Q1, Q3, IQR = Q3 − Q1 을 구하고, 비교 대상 **값**은 **집계 행의 건당(h)** 또는 **유닛별 건당(h)** (같은 분포의 울타리 사용).
- **상단** 기준(과다 투입 관점):

| 상태 | 조건 (건당(h) = v) |
|------|---------------------|
| **정상** | v ≤ Q3 + 1.5×IQR |
| **주의** | Q3 + 1.5×IQR < v ≤ Q3 + 3.0×IQR |
| **이상** | v > Q3 + 3.0×IQR |

- IQR = 0 (표본이 거의 동일)이면 과도한 경고를 피하기 위해 **정상**으로 본다(구현 시 명시).

### 12.4 UI/UX

- **드릴다운**: **대분류** 행을 펼치면 해당 **소분류** 테이블이 나타나는 2단 구조.
- 소분류 행 주요 열: **소분류명**, **누적 건수**, **총 소요시간(h)**, **기준건당(h)(중앙값)**, **상태(신호등)**, **참여 인원**; 필요 시 수동 목표·편차는 보조 패널에 유지.
- **주의·이상**이 붙은 **유닛**을 선택하면 `taskCode`(또는 무코드 1줄)에 대응하는 **필터된 원시 로그**를 **Drawer(Sheet)** 로 일자별로 표시한다.
